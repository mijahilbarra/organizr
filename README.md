# Organizr

React + Express app for discovering repeated Gmail email structures, proposing schemas with Gemini, and saving reusable extractors.

## Local Development

Prerequisites:

- Node.js 20+
- Firebase project with Authentication and Firestore enabled
- Google provider enabled in Firebase Auth
- Gmail API enabled in the related Google Cloud project

Setup:

1. Install root dependencies:
   `npm install`
2. Install Functions dependencies:
   `npm --prefix functions install`
3. Copy `.env.example` to `.env` and fill Firebase, Gemini, and Gmail values.
4. Run locally:
   `npm run dev`

## Firebase Emulator Development

Use this mode to test the Firebase-shaped runtime locally while keeping Firestore in production.

```bash
npm run dev:firebase
```

This starts:

- Firebase Functions Emulator at `http://127.0.0.1:5001/organizr-skip/us-central1/api`
- Vite frontend at `http://localhost:5173`

The Vite dev server proxies `/api/**` to the local Functions Emulator. Do not start the Firestore emulator if you want reads and writes to use the production Firestore database.

You can also run each side separately:

```bash
npm run dev:functions
npm run dev:frontend
```

## Firebase

The Express backend is exported from `server.ts` and exposed to Firebase Functions through `functions/index.ts` as the `api` HTTPS function. Firebase Hosting serves `dist` and rewrites `/api/**` to that Function.

Before deploying:

- Confirm `.firebaserc` points to the intended Firebase project.
- Set `APP_URL` to the Firebase Hosting URL.
- Keep `FIREBASE_FUNCTIONS_REGION` aligned with `firebase.json` and `functions/index.ts`.
- Configure Firebase Auth authorized domains for `localhost` and the deployed domain.

Build checks:

```bash
npm run lint
npm run firebase:build
```

## Custom GPT Actions

The GPT-safe action surface is documented in `functions/openapi.json`. Most orchestration endpoints live under `/api/gpt/**`, and GPT-safe extractor helpers remain under `/api/extractors/**`.

Keep the two OpenAPI files synchronized from the repo root:

```bash
npm run openapi:sync
npm run openapi:check
```

`openapi:sync` rewrites `functions/openapi.json` in canonical JSON format. `openapi:check` fails if that file is non-canonical or contains a media-type object that puts `$ref` directly under `content.application/json` instead of under `schema`.

- `GET /api/gpt/session/capabilities` reports whether the signed-in user has Gmail connected and whether the server has an LLM provider configured.
- `POST /api/gpt/extractor-from-subject` searches Gmail by subject. If Gemini/OpenAI is available, the backend analyzes matching samples and persists an extractor. If only Custom GPT is available, it returns samples plus an `CUSTOM_GPT_ANALYSIS_REQUIRED` response so the GPT can finish the schema and parser itself. The endpoint defaults to `maxResults = 1` unless you ask for more. If `extractorId` is provided, the new subject must be attached to that existing extractor instead of creating a new one.
- `POST /api/gpt/extractor-from-analysis` persists an extractor using schema/script analysis generated by Custom GPT. If `extractorId` is provided, it adds the analyzed subject to that extractor instead of creating a new extractor.

The GPT contract supports provider modes `auto`, `gemini`, `openai`, and `customgpt`. Response payloads also surface `capabilities.providers.customgpt` so the GPT can branch between native analysis and manual analysis without guessing.

Authentication uses a Custom GPT OAuth bridge. The user signs in with Google/Firebase through `/oauth/authorize`; `/oauth/token` exchanges the authorization code for a bearer token that identifies the Organizr user. When Gmail or provider setup is missing, responses include stable action codes, `nextSteps`, and an `actionUrl` pointing the user back to Organizr.

Provider selection in the GPT contract supports `auto`, `gemini`, `openai`, and `customgpt`; `auto` keeps the native provider fallback behavior, while `customgpt` routes the assistant through the manual-analysis branch when Gemini/OpenAI are not available.

6. Si el usuario quiere extender un extractor existente, usa el CRUD de subjects:
